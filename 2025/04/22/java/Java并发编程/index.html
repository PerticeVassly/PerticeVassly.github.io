NEW：初始状态，线程被创建出来但是没有调用start()

死锁的四个必要条件

  1. 互斥条件：该资源任意一个时刻只能由一个线程占用

  2. 请求和保持条件：一个线程因请求资源而阻塞的时候，对于已经获得的资源保持不放

  3. 不剥夺条件：线程已经获得的资源在没有使用完之前不能被其他线程强行剥夺

  4. 循环等待条件：若干线程之间形成一种头尾相接的循环等待资源关系

如何预防死锁：实际上就是破坏上述的四个必要条件

  1. 互斥条件：这个由业务逻辑决定，很难修改

  2. 请求和保持条件：一次性申请所有的资源

  3. 不剥夺条件：一个线程生获取资源的时候如果获取不到，则强行剥夺某一个线程的资源

  4. 循环等待条件：出现循环等待的时候剥夺一方的资源

乐观锁

版本号机制

加入一个数据版本号，表示数据被修改的次数。当准备更新的时候，先检查版本号是否和当前版本号一致，如果一致则更新数据和版本号，如果不一致则说明数据已经被修改过了，更新失败。更新失败则重新操作

CAS算法

然而比较过程本身可能是出现非原子，所以一般要原子操作，对应cmpxchg指令

V：要更新的值

E：期望值

N：新值

其实相当于一个版本号机制，只要当我预期的值（或者说我以为的old value）和当前值一致的时候，就可以更新成功。

当多个线程同时操作一个值，一个时刻只能由一个线程成功，其他线程失败。失败的线程会重新尝试操作，直到成功为止。

Java的CAS是通过cpp + JNI实现的，unsafe包下提供compareAndSwapInt，compareAndSwapLong，compareAndSwapObject等方法来实现的

AtomaticInteger的实现

CAS算法的问题

ABA问题：E和V相同就能说明这个值没有被修改吗？显然不能，可能被修改过了又被改回来了

所以CAS要和版本号机制结合使用，jdk1.5以后由AtomicStampedReference来解决ABA问题

CAS的缺点

CAS实际上多个线程通过自旋转来重试，如果长时间不成功，会给cpu带来巨大开销

而且只能保证一个共享变量的原子操作，所以jdk1.5以后提供了AtomicReference来操作对象而非值，通过把多个变量封装在一个对象中，可以使用AtomaticReference来实现原子操作

ThreadLocal

上面提到了线程实际上在修改“本地内存”，然后同步到主内存中去。但是这种同步对于程序员是透明的，如果不适用volatile关键字，同步何时发生是不可控的

如果想让每个线程都有自己的专属本地变量，可以使用ThreadLocal创建一个threadlocal的时候各个线程都包有一份自己的本地副本，通过get和set方法来修改

为什么要设计ThreadLocal？

实际上是一个共享变量的问题，我想在一个线程内共享这个变量，只能使用类似于“全局变量”的方法，否则就只能使用参数传递了，但是参数传递又会导致代码的可读性和可维护性下降，所以就设计了ThreadLocal来实现线程内共享变量的目的，同时不会影响其他线程的变量

比如说用户连接会话，一般是一个线程对应一个连接会话，这个时候就可以使用ThreadLocal来存储用户的连接会话信息

一种常用方法时设置一个staic的ThreadLocal变量，然后在每个线程中都可以通过ThreadLocal.get()来获取当前线程的连接会话信息

虽然static是唯一一个，但是由于实际上是每个线程都维护了一个map<ThreadLocal, Object>的map，所有Thread共享一个Threadlocal但是不同的线程中取出的值是不同的

ThreadLocal的实现原理

每一个线程中都存在一个ThreadLocalMap的实例，ThreadLocalMap实际上是一个类HashMap，key是ThreadLocal对象hash，value是线程对应的值

当调用ThreadLocal的set的时候，首先通过Thread.currentThread()获取当前线程，获取当前线程的ThreadLocalMap实例，调用map.set(this, value);将自身（作为key）以及值放入这个map中

ThreadLocalMap是TheradLocal的静态内部类

ThreadLocal的内存泄漏问题

上述map.set(this, value)

上述“hashmap”的实现实际上是一个数组 + 链表，数组里实际存储的是Enrty对象，Entry对象里存储了具体的value。但是这个Enrty对象是Extends了WeakReference的

即把传入的ThreadLocal对象（key）是作为一个entry的一个弱引用。弱引用是只要GC扫描到就会被回收（只要Threadlocal对象没有被其他强引用指向），但是value是强引用（成员变量），，和可能依然存在，此时key被回收但是value依然存在，这就导致了内存泄漏的问题

总结下来就是：ThreadLocal的的作用，首先用其地址计算hash作为entry表的index，然后构建entry的时候将其打上弱引用的标记。

  1. Threadlocalmap由于和线程绑定长期存活

  2. 其内部的key 即ThreadLocal对象是弱引用，被回收了, 由于上述“hashmap”实际上是一个entry表，entry是强引用，所以object value也是强引用，所以value不会被回收

使用完以后要调用remove方法，从localmap中移除entry，这样enrty消失了value也就会被回收了

为啥要把Threadlocal设计为弱引用？不把value也设置为弱引用

为什么要把threadlocal设计为弱引用？应为实际上的threadlocal的set和get机制是把自己和当前thread绑定的，并且存储到当前thread的上期文中的。也就是说这个对象的生命周期严格来说是和所有线程绑定的，只有所有用到过这个threadlocal的线程都结束了，这个threadlocal才会被回收。

但是这样很不优雅，也不满足编译优化，应该是如过没有线程持有这个threadlocal了就应该把他gc掉

所以将其设置为弱引用，一旦st1执行完成，除了threadlocalmap的引用，其他地方都没有引用了，但是threadlocalmap是一个弱引用，所以会被gc掉（当然是指所有的thread都没有上述引用了）

为什么value不行？因为value对象仅仅用于初始化，st2执行完成以后，value对象就没有引用了，如果是弱引用机会直接gc。（通俗讲就是编译器不太聪明，他不能判断出st1实际上仍然需要value，他只能看到后面没有直接应用value从而判定其没有引用了）

什么是线程池

当有任务需要处理的时候，直接从线程池中获取线程来处理，处理完后线程不会被立即销毁，而是等待下一个任务再被重新使用

池化技术可以有效减少对象的返回创建和销毁，提高性能和资源利用率，同时也便于管理和监控

线程池的常见参数有哪些，如何解释

ThreadPoolExecutor的参数

corePoolSize：核心线程数，默认情况下，核心线程会一致存活。任务队列没有达到队列容量的时候，最大可以同时运行的线程数

maximumPoolSize：线程池所能容纳的最大线程数，达到这个值后续的新任务会阻塞。任务队列中存放的任务达到任务容量的时候，当前可以同时运行的线程数会增加到maximumPoolSize

workQueue：线程池的execute方法提交的任务会放入这个队列中，等待线程池中的线程来执行。首先判断当前线程数是否小于corePoolSize，如果小于则创建新的线程来执行任务，如果大于corePoolSize，则将任务放入workQueue中等待执行。

keepAliveTime，如果超过该时长，非核心线程就会被回收

threadFactory：线程工厂，创建线程的工厂类

handler：拒绝策略（达到maximumPoolSize）

综上流程就是：

任务提交到qeue中，如果当前运行的线程小于corePoolSize，则创建新的线程来执行任务；如果当前运行的线程大于corePoolSize，则将任务放入workQueue中等待执行；如果workQueue满了，此时只要当前运行的线程小于maximumPoolSize，则创建新的线程来执行任务；如果当前运行的线程大于maximumPoolSize，则执行拒绝策略

每个线程任务执行完后，会从任务队列中获取下一个任务来执行，如果设置了超时时间，如果长时间没取到任务，则会被回收（核心线程则不会）

可以看出当队列满了以后，线程池就会“扩容”，当任务少了以后，任务队列中没有任务了，多余的线程就会被回收，线程池就会“缩容”

线程池核心线程会被回收吗

一般不会，如果指定了allowCoreThreadTimeOut = true，则会被根据设定的keepAliveTime回收

核心线程的状态

java线程六状态中有block，waiting， timed_waiting这样几种等待状态（new， runnable，terminated）

如果没有设置存活时间，则一致处于waiting（这和waiting状态的定义也是符合的，等待别人唤醒）

设置了，则会退出并且进入TERMINATED状态，从工作线程集合中移除

线程池拒绝策略

ThreadPoolExecutor.AbortPolicy：抛出RejectedExecutionException异常

ThreadPoolExecutor.CallerRunsPolicy：调用者线程中直接执行，如果调用者线程已经结束，则直接抛出异常

ThreadPoolExecutor.DiscardPolicy：丢弃任务，不抛出异常

ThreadPoolExecutor.DiscardOldestPolicy：丢弃队列中最旧的任务，然后尝试重新执行

CallerRunsPolicy拒绝策略有声明风险，如何解决

风险在于可能会导致调用者线程阻塞，但是一旦调用者阻塞了，线程池的fetch任务等等管理操作也会随之阻塞，导致线程池后续任务无法及时执行

如何解决？增加blockingqueue的容量，增加maximumPoolSize的值，增加corePoolSize的值提高任务处理效率

线程池常用的阻塞队列有哪些？

LinkedBlockingQueue：主要用于FixedThreadPoolExecutor和SingleThreadPool，队列大小为Integer.MAX_VALUE，这两个线程池都是核心线程数等于最大线程数，SingleThreadPool二者都是1

SychronousQueue：主要用于CachedThreadPoolExecutor，队列大小为0，线程池的最大线程数是Integer.MAX_VALUE, 不存储任务，仅仅只是传递，提供同步功能

DelayedWorkQueue：数字要用于ScheduledThreadPool和SingleThreadScheduledPool，按照延迟时间长短堆任务排序提供，且自动扩容，最大可到达Integer.MAX_VALUE

ArrayBlockingQueue：数组实现，一旦创建，不能修改

线程池中线程异常，销毁还是复用

execute提交，如果异常没有被捕获，当前线程终止，异常信息打印到控制台或者log中，线程池创建一个新的线程来替换他

submit提交，异常封装在submit返回的Future对象，调用get的时候返回，此时线程不会因为异常终止，而是继续执行下一个任务

线程池的大小设计

太小有大量任务在排队，导致响应时间变长

太大，线程切换频繁，导致cpu上下文切换开销大，响应时间变长

针对cpu计算密集型和i/o密集型的线程池大小设计，有以下共识

cpu计算密集型：cpu核心数+1

I/O密集型：cpu核心数*2

PriorityBlockingQueue：可以说看成是一个线程安全的PriorityQueue，且支持阻塞操作

为了防止OOM（毕竟无界），可以继承这个class并且重写offer方法，限制队列的大小

Future

核心思想是异步调用

主要包括下面四个功能

  1. 取消任务

  2. 判断任务是否被取消

  3. 判断任务是否已经执行完成

  4. 获取任务执行结果

FutureTask提供了Future接口的基本实现，用于封装Callable和Runnable对象，ExecutorService的submit方法返回的就是FutureTask对象

Runnable接口 + Future接口 = RunnableFuture接口

futuretask就是一个runnablefuture接口的实现类

Future + CompletionStage = CompletableFuture类

为什么需要设计一个completeableFuture类？

因为futuretask不支持异步任务的编排，一旦调用get方法就会阻塞当前线程，无法实现异步编排的目的

completableFuture类实现了CompletionStage接口，支持异步编排的功能，还提供了函数式编程等等能力

CompletableFuture 如何处理异常

whenComplete方法在任务完成时触发回调函数

exceotionally方法处理异常并且重新抛出

handle方法处理异常并且返回一个新的值

AQS

AbstractQueuedSynchronizer, 抽象队列同步器

提供了一个通用框架，用于实现各种同步器，例如可重入锁ReentrantLock，信号量Semaphore，倒计时器CountDownLatch，等等

有点像信号量的实现

核心思想：

如果被请求的共享资源空闲，则讲当前请求资源的线程设置为有效的工作线程，共享资源锁定。如果共享资源被占用，需要线程等待和唤醒机制

CLH锁

对于自旋锁进行改进，基于单链表，多线程的时候，把请求锁的线程组织成一个单向队列，每个等待的线程自旋访问前一个线程的状态，前一个线程释放锁，后一个线程获取锁

AQS对其进行了变体

纯自旋消耗大量cpu资源，改为自旋 + 阻塞

由于引入了阻塞唤醒操作，前面的线程释放锁，要对后面的线程进行唤醒操作，新增next指针，形成双向队列

Semaphore

信号量

允许多个线程同时访问共享资源

主要是acuqire和release方法来实现的

也有公平锁和非公平锁的实现

CountDownLatch

但是countDownLatch允许当前线程等待其他线程执行某一操作完毕（即conntdown操作），或者说是等待其他线程的状态改变，当调用countDown方法的时候state会减一，当state为0的时候，当前线程可以继续执行

你可以理解为一种支持计数的join方法

当然使用CompleteableFuture也可以实现类似的功能

CyclicBarrier有什么作用

类似上述CountDownLatch

设置一个屏障，当所有线程都到达这个屏障的时候，所有线程一起继续执行

实际上也是维护一个计数器，当所有线程都到达这个计数器的时候，计数器归零，此时执行构造方法中输入的任务
