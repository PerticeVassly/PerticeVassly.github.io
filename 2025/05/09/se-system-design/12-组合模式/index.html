组合模式

组合模式 （Composite Pattern）

组合多个对象形成树型接口以标识“整体-部分”的结构层次。组合模式对当个对象和组合对象的使用具有一致性

又可以称之为“整体-部分”（Part-Whole）模式，属于对象的结构模式，它将对象组织到树结构中，描述整体和部分的关系

Composite Pattern: Compose objects into tree structures to represent part-whole hierarchies. Composite lets clients treat individual objects and compositions of objects uniformly.

模式结构

classDiagram
    class Client {
    }

    class Component {
      + operation()
      + add(Component c)
      + remove(Component c)
      + getChild(int i)
    }

    class Leaf {
      + operation()
    }

    class Composite {
      + operation()
      + add(Component c)
      + remove(Component c)
      + getChild(int i)
    }

  Component <|-- Leaf
  Component <|-- Composite
  Component <--o Composite : children
  Client --> Component

其中operation的实现为

包括如下角色

Component：抽象构件

Leaf：叶子构件

Composite：容器构件

Client：客户端

对于任意一个构件，其只知道自己是leaf还是composite，其下的所有构件都是Component类型的对象。

但是这样似乎会带来一个问题就是Composite类的add、remove、getChild方法在leaf被暴露了。此时leaf中的这些方法应该有报错等等信息。即只有operation方法是依次递归的。

与传统递归不同，这样递归使用到了多态的特性，将leaf情况的代码封装到了具体的leaf class中，降低了耦合

优缺点

优点

清楚定义分层次的复杂对象，标识对象的全部或者部分层次，使得增加新构建也很容易

客户端调用简单，客户端可以一致的使用组合结构或者其中的单个对象

可以形成比较复杂的树形结构

可以比较容易在组合体内加入对象构建

缺点

设计变得抽象，不是所有方法都与叶子对象子类都有关联（上述方法暴露的问题）

难以对容器中的构建类型进行限制，毕竟只要实现Component接口就可以了

适用环境

需要表示一个对象整体或者部分层次

客户端可以针对抽象构件编程，无需关心对象层次结构的细节

对象的结构是动态的并且复杂程度不一样，但是客户端需要一致处理他们

模式应用

xml是一个很好的例子，xml的节点可以是一个节点，也可以是一个节点的集合

模式扩展

更复杂的组合模式

组合模式其实利用了多态来实现递归，其实可以有多种形式比如我多个具体类继承Component类作为leaf/Composite类，但是这样实际上可以再提取出一个公共的抽象类，或者接口来实现即leaf接口和Composite接口/抽象类

透明的组合模式

即所有接口全部放到最顶层的Component类中，客户端可以调用

安全组合模式把Composite类中的add、remove、getChild方法都放到Component类中，Component类不暴露这些方法。好处是不会出现调用leaf的add、remove、getChild方法的情况，坏处是客户端基于Component类编程的时候无权构建和修改，只能在Composite类中构建和修改
