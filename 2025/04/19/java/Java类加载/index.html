java类加载

分为加载，链接，初始化三个阶段。

其中连接又分为验证，准备，解析三个阶段。

加载：通过全类名获取类的二进制字节流

把字节流中的数据转换为方法区的运行时数据结构

在内存中生成一个代表这个类的Class对象，作为方法区这些数据的访问入口

类加载器

是一个负责加载类的对象，根据类名查找文件系统中的字节码文件，读取字节码文件，解析字节码文件，生成Class对象。

但是这样就有一个先有鸡还是先有蛋的问题了。

实际上基本的ClassLoader是由JVM内置的

  1. Bootstrap ClassLoader：引导类加载器，负责加载Java核心类库（rt.jar等等，以及-xbootclasspath指定的类库）

  2. Extension ClassLoader：扩展类加载器，负责加载Java的扩展类库（%JRE_HOME%/lib/ext）,以及被java.ext.dirs指定的类库

  3. AppClassLoader：应用类加载器，负责加载用户类路径（classpath）下的类

除了基本加载器，也可以使用自定义加载器进行扩展，满足自己的特殊需求（比如加密）

当我们对一个classLoader调用getParent()方法时，可以一层一层逐步向上查找，直到Bootstrap ClassLoader为止。

如果一个类的加载器是null，说明这个类是由Bootstrap ClassLoader加载的。（eg：Extension ClassLoader的父类都是Bootstrap ClassLoader，所以Extension ClassLoader的父类是null）

双亲委派机制

如何判断一个class到底应该被哪一个类加载器加载？

每个classloader实例都有一个相关的父类加载器，需要查找类或者资源的时候，classloader会在试图亲自查找类或者资源之前，把搜索类或者资源的任务委托给其父类加载器，如果父类加载器为null，调用Bootstrap ClassLoader，如果父类加载器无法加载这个类或者资资源，调用自己的findClass方法

也就是说，每个加载请求都会到达最顶层的Bootstrap ClassLoader，然后一层一层判断自己是否能加载，直到重新返回到最底层的加载器

ps: java判断类的全名是否相同，还要加载此类的类加载器是否相同。

因为这样可以确保核心类总是由Bootstrap ClassLoader加载，避免了核心类被用户自定义的类加载器覆盖。
