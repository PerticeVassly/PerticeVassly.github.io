seac2-demandanalysis

需求分析基础

(1) 建立分析模型，达成开发者和用户对于需求信息的共同理解

(2) 依据共同的理解，发挥创造性，创建软件系统解决方案

需求分析模型的特点以及常见的需求分析模型

需求分析模型一方面使用了软件的内部实体，以类，对象等等属性作为模型的基本元素，另一方面使用了问题域内的概念来描述软件内部实体

常见的需求分析模型

结构化方法

数据流图

将系统看成过程的集合，其中一些由人来执行，另外一些由软件来执行。执行的过程就是对数据的处理，接受数据输入进行转换输出结果

DFD的基本模型元素有四种

外部实体(external entity)

过程(process)

数据流(data flow)

数据存储(data store)

一些语法规则

（1）有输入则一定要有输出，而且输入数据集和输出数据应该不一样

（2）数据流一定适合过程产生联系的，要么是过程的数据输入，要么是过程的数据输出

每个对象应该都有一个可以唯一标识子集的名称，过程用动词，外部实体，数据流和数据存储用名词

三个层次类别:

上下文图(context diagram)

0层图(level-0 diagram)

N层图(level-N diagram)

(1) 上下文图：是系统功能的最高抽象。将整个系统看作一个过程，整个过程实现系统的所有功能。上下文图中存在且仅存在一个过程，表示整个系统记为0

(2) 0层图，位于上下文图中的下面一层就是0层图。是上下文图中单一过程的细节描述，是对单一过程的第一次功能分解。需要在一个图中概括系统的所有功能

(3) N层图

0层图的每一个过程都可以进行分解以展示更多的细节。

低于0层图的子图上通常不显示外部实体。夫过程的输入输出数据流称之为子图的接口流，在子图中从空白区域引出。

实体关系图

实体关系图使用实体属性和关系三个基本元素来描述数据模型

最大基数和最小基数：

最大基数：对于关系中任意的其他实体实例，该实体可能参与关系的最大数量。

同理可得最小基数。

属性是可以对实体进行描述的特征。

如果可以用一个或者多个属性唯一标识一个实例，则这些属性或者属性组合就称之为实体的标识符(identify)或者键(key)

面向对象分析

统一建模语言(Unified Modeling Language UML)是面向对象分析的主要模型基数。

可以分为用例图，交互图，状态图和对象约束语言OCL

用例图

用例时软件需求的组织或者时表示的方法之一

每个用例是对相关场景的集合，这些场景是用户和系统之间的交互行为序列，帮助实现用户的目的

基本元素：用例，参与者，关系，系统边界

参与者：代表的是可以同系统进行交互的角色。但是其实不一定要是人,也不一定是单个人

  • An actor is a role that a user or other system plays with respect to the system to be developed.

  • A single actor in a use case diagram can represent multiple users (orsystems) .

  • A single user (or system) also may play multiple roles.

  • Actors don‘t need to be human, e.g., an external system that needs someinformation from the current system is also an actor.

用例粒度合适的判断标准是：用例描述了为了应对一个业务实践，由一个用户发起，并且在一个连续时间内完成，可以增加业务价值的任务

注意：不要将粒度细化得过小，不要把用例细化为单个操作。应为单个得操作(增删改查)是没有业务价值的

不要把同一个业务目标细化为不同的用例

不要将没有业务价值的内容作为用例。

( 用例描述模板 )

类图(领域模型)

基本元素：对象，类，链接，关联(聚合)，继承

对象：包含标识符，状态，行为，类

链接：系统中的对象不是孤立存在的，他们需要相互写作完成任务。对象这种相互写作的关系称之为链接

关联：指出了类之间的某种语义关系。是对链接的抽象

  • 类中有一种特殊的关系称之为聚合(aggregation)

  • 如果整体除了包含部分之外还对部分拥有完全的管理职责，一旦一个部分属于整个整体其就不可能同时属于其他整体也无法单独存在，则称之为组合(composition)

继承：泛化关系

  • Organise the domain object classes into a hierarchy

  • Classes at the top of the hierarchy reflect the common features of all classes

  • Object classes inherit their attributes and services from one or more

  • super-classes. these may then be specialised as necessary

基本流程

  1. 对于每个用例文本,尤其是场景描述,建立局部的概念类图

  2. 将所有用例产生的局部概念类图进行合并,建立软件系统的整体概念图

如何建立局部的概念图

(1) 识别候选类

(2) 确定概念类：确定一个候选类是否是概念类的准则是：根据系统的需求，该类的对象实例和状态和行为是否全部必要

  • 1）如果候选类的对象实例急需要维护一定的状态由需要以及状态表现一定的行为那么其可以被确定为一个概念类

  • 2）如果候选类的对象实例只需要维护状态，不需要表现行为则其就不需要被确定为概念类而是其他概念类的属性

  • 3）如果候选类的实例对象不需要维护状态但是需要表现行为，则说明需求由纰漏或者要将其行为转到由支持能力的其他概念类上

    1. 既不需要维护状态又不需要表现行为,则该类应该被完全剔除

(3) 识别关联

  • 分析⽤例⽂本描述，发现概念类之间的协作需要协作的类之间需要建⽴关联。

  • 分析和补充问题域内的关系

  • 例如概念类之间的整体部分关系和明显的语义联系。

  • 对问题域关系的补充要适可⽽⽌，不要把关系搞得过度复杂化。

  • 去除冗余关联和导出关联。

(4) 识别重要信息

交互图(顺序图)

对象需要相互协作才能完成任务,交互图就是描述对象协作的技术.

  1. 简单顺序图

  1. 系统顺序图

状态图

状态图的简单元素包括状态,开始状态,结束状态,实践,监护条件,活动和转换

  1. 确定上下文环境

  2. 识别状态

  3. 建立状态转换

  4. 补充详细信息
