工厂模式

简单工厂模式 (Simple Factory Pattern)

模式定义

又称之为静态工厂方法（Static Factory Method）模式，其属于类创建型模式。在简单工厂模式中，可以根据参数的不同返回不同类的实例。简单工厂模式专门定义一个类来负责创建其他类的实例，被创建的实例通常都有共同的父类

模式结构

classDiagram
    class Factory{
        +createProduct() Product
    }
    class Product{
        +operation()
    }
    Product <|-- ConcreteProductA
    Product <|-- ConcreteProductB
    Factory ..> ConcreteProductA
    Factory ..> ConcreteProductB

包含如下角色

Factory: 工厂类，负责创建产品类的实例

Product: 抽象产品类

ConcreteProduct: 具体产品类

模式分析

将对象的创建和对象本身业务处理分离可以降低系统的耦合度

工厂方法是静态方法

问题在于工厂类的职责相对过重，增加新的产品需要修改工厂类的判断逻辑，违背了开闭原则

简单工厂模式的要点在于：当你需要什么，只需要传入参数就可以获取你需要的对象，而无需知道其创建细节

优缺点

优点：

  • 实现了对于责任的分割，提供了专门的工厂类用于创建对象

  • 客户端无需知道所创建的具体产品的类名，只要知道对象的参数即可

  • 通过引入配置文件，可以在不修改任何客户端代码的情况下更换和增加新的具体类

缺点：

  • 工厂类集中了所有产品的创建逻辑, 臃肿也不符合开闭逻辑

  • 使用简单工厂会增加系统中类的个数，增加系统的复杂度和理解难度

  • 系统扩展困难，一旦添加新产品就不得不修改工厂类的判断逻辑，产品类型较多的时候，工厂类会变得很复杂

  • 工厂角色无法形成基于继承的等级结构

使用场景

工厂类负责创建的对象比较少

客户端不关系如何创建对象

模式扩展

上述方法创建了一个单独的工厂类，而且使用静态方法生产产品。同时我们又创建了一个抽象产品类，自然可以把这个静态方法写到抽象产品类中，这是一样的。称之为对简单工厂模式的简化

工厂方法模式 (（Factory Method Pattern）

模式定义

工厂方法模式（Factory Method Pattern）又称为工厂模式，也叫虚拟构造器模式（Virtual Constructor）模式或者多态工厂模式（Polymorphic Factory Pattern）。

实际上是将产品类的实例化操作延迟到工厂的子类中完成

实际上是对if-else的进一步拆分（每一次拆分就引入一个抽象层，使用抽象来替代if-else）

模式结构

classDiagram
    class Factory{
        +createProduct()  Product
    }
    class Product{
        +operation()
    }
    Product <|-- ConcreteProductA
    Product <|-- ConcreteProductB
    Factory <|-- ConcreteFactoryA
    Factory <|-- ConcreteFactoryB
    ConcreteFactoryA ..> ConcreteProductA : create
    ConcreteFactoryB ..> ConcreteProductB : create

模式分析

核心的工厂类不再负责所有的产品的创建，而是将具体的创建工作交给子类去做

可以允许系统在不修改工厂角色的情况下引进新产品

简单工厂是将所有的创建逻辑都放在一个工厂类中，可以认为这种方式的封装粒度还是过大了，同时以传入参数的形式来决定创建逻辑本身就必然导致代码的修改。要解决这个问题只能将选择逻辑放在客户端中————即类的选择来替代这个参数选择，同时这个类本身也作为封装封装了创建对象的复杂逻辑。但是缺陷也是明显的————引入了大量的类

很好符合了开闭原则，退化以后变成简单工厂

很好使用了依赖倒转原则，使用抽象产品类以及本身继承抽象工厂类，实际使用只要更换对应工厂即可

实际应用开发中一般将工厂类的实例化过程进行改造，而不使用new关键词来创建对象，而是将具体类的类名写入配置文件中，再通过Java的反射机制，读取XML格式的配置文件，根据存储在XML文件中的类名字符串生成对象。

优缺点

优点：

用户只需要关心所需产品对应的工厂，无需关心创建细节，甚至无需知道具体产品类的类名

工厂剋以自主决定创建何种产品对象，以及创建这个对象的细节可以完全封装在具体的工厂内部

系统中加入新产品的时候，无需修改抽象工厂和抽象产品提供的接口，只需要添加具体的工厂和具体的产品即可

缺点：

需要编写新的具体产品类，而且还要提供与之对应的具体工厂类，系统中的类的个数将成对增加

增加了系统的抽象性和理解难度

使用场景

一个类不知道它所需要的对象的具体类

一个类通过子类来指定创建哪一个对象

将创建对象的任务委托给多个工厂子类中的某一个，客户端在使用的时候无需关系是哪一个工厂子类创建产品类，需要时再动态指定

抽象工厂模式（Abstract Factory Pattern）

模式定义

抽象工厂模式（Abstract Factory Pattern）提供一个创建一系列相关或相互依赖对象的接口，而无需指定它们的具体类。抽象工厂模式又称之为Kit模式，属于对象创建型模式

可以视为在工厂方法中添加了一个新的产品类————增加了一个新的产品结构（依然从属于一个工厂）

模式结构

classDiagram
    class AbstractFactory{
        <<interface>>
        +createProductA()  ProductA
        +createProductB()  ProductB
    }
    class ProductA{
        <<interface>>
        +operation()
    }
    class ProductB{
        <<interface>>
        +operation()
    }
    AbstractFactory <|.. ConcreteFactoryA
    AbstractFactory <|.. ConcreteFactoryB
    ProductA <|.. ConcreteProductA1
    ProductA <|.. ConcreteProductA2
    ProductB <|.. ConcreteProductB1
    ProductB <|.. ConcreteProductB2
    ConcreteFactoryA ..> ConcreteProductA1 : create
    ConcreteFactoryA ..> ConcreteProductB1 : create
    ConcreteFactoryB ..> ConcreteProductA2 : create
    ConcreteFactoryB ..> ConcreteProductB2 : create

抽象工厂模式包括

AbstractFactory: 抽象工厂类

ConcreteFactory: 具体工厂类

AbstractProduct: 抽象产品类

Product: 具体产品类

优缺点

优点

隔离了具体类的生成

保证客户端始终只使用一个产品族中的产品

符合开闭原则

缺点

难以扩展抽象工厂来生产新种类的产品，产品簇很容易扩展，但是新产品很难

开闭原则的倾斜性：新增新的工厂和产品族容易，增加新的产品等级结构麻烦

使用场景

系统不应当依赖产品实例的创建，组合和表达的细节

同一个产品族的产品在一起使用

所有产品以同样的接口出现，客户端不依赖于具体实现
