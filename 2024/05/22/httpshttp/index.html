加密传输和加密存储

传输

HTTP

采用明文传输的形式，其中GET方法的请求参数会在URL中显示，留存在浏览器缓存中，容易被窃取。尽管post可以将参数放入请求体(body)中传输，但是由于没有直接采用加密，信息仍然可能被抓包后获取。

HTTPS

使用SSL/TLS完成加密和验证。

  1. 客户端向服务器发送请求。

  2. 服务器受到请求后将自己的公钥Pub发送给客户端

  3. 客户端使用公钥加密一个随机key，发送给客户端，客户端解密后得到这个随机key（此随机key就作为一个对称密钥）

  4. 接下来服务器给客户端发送的信息全部都用这个key加密

  5. 客户端使用这个key解密，得到消息

这样就实现了传输加密。

ps:之所以使用对称加密（key加密解密）是从性能考虑的，两组非对称加密的耗时远大于对称加密。

http中间人攻击

但是上述操作并不能确定接收到的公钥到底是不是真的由服务器发送过来的。

如果客户端收到了一个伪造的公钥（攻击者有自己的密钥）（攻击者截获真正的服务器公钥），就可以伪装成服务器与客户端通信，再伪装成客户端与服务器通信。（HTTP中间人）从而获取所有信息。

因此需要一种方法可以鉴别发送的公钥是可信的（鉴别其没有被修改）

解决方法是CA证书+数字签名。

CA机构是公信机构，因此浏览器或者系统会内置其证书和公钥信息。

服务器再最开始的时候不在仅仅发送公钥，而是发送一个CA证书+数字签名。通过将CA证书（text部分）做Hash得到的值利用CA的私钥加密，可以得到一个数字签名。而客户端在拿到传输过来的证书和数字签名后，只需要对CA证书做相同Hash（方法一般在证书内已经指定了），再对数字签名用公钥解签，二者进行比对，就可以判断证书内容有没有被修改了。由于CA私钥是难以获取的，攻击者不可能为修改过的text的Hash，伪造一份加数字签名使其在被公钥解密后仍然保持原来的值。

所以计算机内的CA机构本身的证书以及其公钥是十分重要的。CA机构是以信任链的形式运行的，如果用户信任了有风险的CA（配置了新的根证书），或者CA机构的私钥泄露，皆会造成上述加密的失效。

但是这样会引申出两个问题。首先是用户如何获取CA的地址？一般是预装在系统和浏览器中。其次CA显然不可能为所有的网站都签发证书，此时其会授权中间CA为其进行签发，形成一个证书链。因此用户在收到证书后，如果显示是中间CA签发的证书，还需要再对这个中间CA自身的证书进行验证（检查其公钥是否经过了签发）

完整示意图

S -> C: 自身公钥（Internal-CA签发）

C 请求获取Internal-CA的公钥

Internal-CA -> C: Internal-CA的公钥（Root-CA签发）

C 请求获取Root-CA的公钥

Root-CA -> C: Root-CA的公钥（内置在系统中）

C 确认Root-CA的公钥

C 确认Internal-CA的公钥

C 确认S的公钥

C -> S: 用S的公钥加密的随机key

S -> C: 用随机key即为对称密钥加密
开始通信

储存

由于上述加密操作，以及在信道中间点进行窃听和截取的困难性，对于信源和信端的攻击往往收效更大。

明文存储

为了防止数据库被拖库后造成的信息泄露，隐私信息（密码）在数据库中一般不以明文存储。

MD5或者其他Hash算法存储

用户输入的密码进行MD5（举例）后发送给服务器，服务器进行比对。这样即使服务器数据泄露，攻击者也很难伪造出一个密码，使得其MD5加密后和服务器中储存的密码对应。

但是大多数人的密码是常见的排列组合，而常见的加密算法（如MD5）又是固定的，所以可以为一些常见密码组合构建一个彩虹表（rainbow table），从而通过数据库中的密码摘要直接对比得出原始密码，从而实现破解。由于彩虹表只要计算一次就可以反复使用，目前针对各类加密算法已经形成了大量的彩虹表（还在快速增加）这种加密方式安全性较低。

加盐后Hash存储

在密码中中固定的位置插入特定的字符串，使其不再是人常用的组合，在进行Hash存储到数据库中，数据库同时存储salt的值和MD5值，攻击者同样难以针对MD5值伪造一个密码使其加盐之后能对应这个MD5值（更何况加盐的方式还是未知的）同时还使得彩虹表难以使用。

为了加盐的效果：salt要足够长且随机。
