Java语言特点

OO

平台无关 (跨平台已经是上古时代的优点了，Java的强大之处在于其生态)

支持多线程(cpp没有内置的多线程，必须调用os的多线程，java提供了多线程支持)

可靠性（异常处理和自动内存管理）

高效性（JIT等技术优化）

支持网络编程

编译与解释并存。

Jave SE， Java EE

SE (Standard Edition): 提供了核心类库以及虚拟机

EE (Enterprise Edition): 提供了企业级应用的开发框架和API，主要用于Web应用和企业级应用的开发。（包含Servlet， JSP, EJB, JDBC, JPA, JTA, JavaMail, JMS）

JVM, JDK, JRE

JVM: 负责解释执行生成出来的字节码

JDK(Java Develop Kit)：功能齐全的Java开发工具包，包含JRE, Javac, javadoc, jdb, jconsole, javap等工具

JRE(Java Runtime Environment)：Java运行环境，包含JVM和Java类库。

字节码

面向虚拟机

.java -> javac -> .class -> 解释器 & JIT -> 机器码

编译与解释并存

编译部分

Javac编译器：将Java源代码编译为字节码（.class文件）。

JIT(Just In Time)编译器：在运行时将字节码编译为机器码，提升性能。

AOT(Ahead Of Time)编译器：在运行前将Java源代码编译为机器码，避免了JIT预热等等方面的开销。AOT还能减少内存占用和增强Java程序的安全性（不容易被反编译）

解释部分

JVM：Java虚拟机，负责执行字节码和JIT编译后的机器码。

Oracle JDK vs OpenJDK

Oracle JDK是Oracle公司提供的非开源（限时免费）Java开发工具包，包含了OpenJDK的所有功能，并且提供了一些额外的商业特性和支持。

OpenJDK是Java的开源实现，包含了Java SE的核心功能和API。

Java和c++的区别

Java不提供指针

Java类单继承，Cpp支持多继承，Java类不可以多重继承，但是可以实现多个接口

Java是GC，cpp使用RAII/手动管理

Java支持方法重载，但是Cpp支持运算符重载

一些不太常用的Java关键字

native： 表示方法是用其他语言（如C/C++）实现的，通常用于JNI（Java Native Interface）。

throw: 表示在代码中实际抛出一个异常对象

throws: 用于声明异常可能被抛出

位移运算符

eg: >>>: 逻辑右移，符号位补零

其应用有：位字段管理，哈希算法和加密解密，数据压缩，数据校验，内存对齐

如果出现 >> 32 实际上是 >> (32 % 32) long 同理

包装类型的缓存机制

Byte Short Integer Long的四种包装类型默认创建了数值 [-128, 127] 的相应类型的缓存数据， Character创建了数值在[0, 127]范围内的缓存数据， Boolean直接缓存true和false两个值。

除非你显式new一个对象，否则自动装箱等机制会优先使用缓存的对象

注意：Java中==比较的是对象的内存地址(String除外, 被编译器重载了运算符), HashSet中首先比较HashCode，再比较equals。如果你没有override这两个方法那么就会以引用比较，然而Integer对象的hashCode方法是直接返回对象的值，equals方法也是比较对象的值。

浮点数精度丢失

使用BigDecimal。 注意：BigDecimal中的等值比较应该使用compareTo方法，而不是使用equals方法。因为equals方法还会比较数值和精度，而compareTo方法只比较数值

静态变量有什么用

可以被所有实例共享。静态变量只会被分配一次内存

字符型常量和字符串常量

char在java中占两个字节，字符串常量实际上是一个地址

浅拷贝，深拷贝，引用拷贝

实现Cloneable接口的类可以使用Object类中的clone()方法来实现对象的拷贝。否则不可以使用（尽管是protected且所有类都继承自这个类）

super.clone()方法是Object类中的一个protected方法，返回一个对象的浅拷贝。成员变量内部引用类型的对象仍然指向原对象的内存地址。其中Clone整个方法只是一个方法标记，由编译器生成具体方法

String Builder和 String Buffer

StringBuilder是非线程安全的，性能更好，适用于单线程环境。

StringBuffer对于方法加了同步锁或者对于调用的方法加了同步锁

总结：

操作少量的数据： 适用String

单线程操作字符串缓冲区下操作大量数据；适用于StringBuilder

多线程操纵字符串缓冲区下操作大量数据：适用于StringBuffer

字符串拼接 += +

所谓的 += 实际上是通过StringBuilder调用append() 方法实现的，拼接完成以后调用toString()得到一个String对象。 如果滥用 += + 会导致创建过多的StringBuilder对象

常量池

常量池(const pool)实际上是.class文件中的一个表，存放了编译器生成的常量值,包括Unicode字符，字符串，int double等字面值，对于各种接口方法字段的符号引用。

运行时常量池(runtime constant pool)是JVM在加载类时创建的一个常量池，包含了类的字面量和符号引用。运行时常量池也是每一个类都有一个，jvm为这些常量创建一个对应的实例对象，并且加入全局表中（存对应实例的引用）（例如String Pool），后续需要使用到常量（的引用）的时候，就从这些全局表中直接获取了。

String s1 = new String(“abc”)

首先会创建常量值对象"abc"，然后创建一个新的String对象，指向常量值对象"abc"的内存地址。(如果常量池中已经存在"abc"，则直接指向常量池中的对象)

Java异常类

Throwable

  • Error: JVM错误，通常无法处理（如OutOfMemoryError）
  • Exception: 异常
    • Checked Exception: 受检查异常必须处理
      • IOException: 输入输出异常
      • ClassNotFoundException: 类未找到异常
      • SQLException: SQL异常
      • FileNotFoundException: 文件未找到异常
      • …
    • Unchecked Exception: 不受检查异常可以不处理
      • ArithmeticException: 算术异常（如除以零）
      • ClassCastException: 类型转换异常
      • IllegalThreadStateException: 非法线程状态异常
      • IndexOutOfBoundsException: 索引越界异常

Checked Exception vs Unchecked Exception

如果受检查异常没有被catch或者throws，编译器会报错。例如适用FileInputStream打开一个文件，由于其内部throw了exception，因此你的代码中也必须要catch或者throws这个异常

不受检查异常不需要强制处理。例如你不需要显式处理数组越界异常，java会自动处理这个异常，当然你也可以catch这个异常

try-with-resources vs try-catch-finally

适用范围，任何实现java.lang.AutoCloseable或者java.io.Closeable接口的对象

在try-with-resources中任何catch或者finally块在声明的资源关闭后运行

异常抛出注意事项

不要把异常定义为静态变量

尽量抛出更加具体的异常

避免重复记录日志，如果子捕获异常的地方已经记录了足够的信息，那么当业务代码中在再次抛出这个异常的时候，就不应该再次记录相同的错误信息

泛型的几种方式

泛型类，泛型接口，泛型方法

UnSafe

Java中提供了一个Unsafe类，提供了一些底层操作的API，例如直接内存访问，CAS操作，线程挂起和唤醒等，直接获取对象的内存地址，直接获取offset操作对象

在getUnSafe()中，会对调用者的classLoader进行检查，判断当前类是否由BootStrapClassLoader加载，如果不是的话就会抛出一个SecurityException。避免了随意调用

如何解决？1. 使用反射获取。2. java -Xbootclasspath/a: ${path} // 其中path为调用Unsafe相关方法的类所在jar包路径把调用UnSafe的相关方法追加到默认的bootStrap路径中

以下是其中的一些方法

put，putOrder, putVolatile

put是直接put

putOrdered...保证有序性, 但是你不一定能看到这个值

putVolatile...保证可见性，有序性

顺序写时加入的内存屏障是StoreStore类型，而volatile写入时加入的内存屏障是StoredLoad类型

前者保证首先值立即刷新到内存且先于后面的store指令执行

后者保证首先值立即刷新到内存且先于后面的load/store指令执行，且只有之前的所有指令都完成以后才能执行后面的指令

allocateInstance

提供了一种跳过构造器的方式来创建对象实例。

CAS

Compare And Swap， 是一种乐观锁的实现

如果预期原值和实际原值相同，则将新值写入内存地址并返回

实际上是cpu底层执行cmpxchg

线程

UnSafe类中提供了park，unpark，monitorEnter，monitorExit等方法来实现线程的挂起和唤醒

内存屏障

强制这个值读取或者写入到主内存中，应为java实际上可能仅仅只是修改/读取了线程的本地内存，具体何时同步未知。

SPI

Service Provider Interface

Application Programming Interface

调用方 -> [接口 <- 实现方] api

[调用方 -> 接口] <- 实现方 spi

区别在于接口规则由谁制定，API是调用方制定的，SPI是实现方制定的

提供方在Meta-INF/services/下创建一个文件，文件名为接口的全限定名，文件内容为实现类的全限定名, 再根据反射加载实现类，当然具体实现类是要放java的classpath下

缺点

  • 需要遍历加载所有的实现类，不能做到按需加载，这样效率还是相对较低的。
  • 当多个 ServiceLoader 同时 load 时，会有并发问题。

序列化与反序列化

场景：持久化Java对象或者将Java对象保存在文件中，或者在网络传输Java对象

transient关键字：标记不需要序列化的字段（只能修饰变量）

常见的序列化协议有哪些

JDK自带的序列化方式一般不会用，应为其效率低而且有安全问题

如果要使用JDK自带的序列化，只需要实现java.io.Serializable接口即可

jvm在序列化和反序列化的时候会检查serialVersionUID（根据类的结构信息），如果没有定义，jvm会自动生成一个，但是如果类的结构发生了变化，serialVersionID就会不同，从而导致失败。当然你也可以直接强行设定ID相同从而绕过jvm检查；为什么serialVersionUID明明是静态变量也会被持久化，因此serialVersion是一个特列（），实际上只要有一个static final long 且名字为serialVersionUID就可以被jvm自动识别了

常用的有 Hessian，Kryo，Protobuf，ProtoStuff

JSON和XML也可以序列化，可读性好但是性能差

IO流

主要是从四个抽象基类中派生出来的

InputStream / Reader：前者是字节输出流，后者是字符输出流

OuputStream / Writer：前者是字节输出流，后者是字符输出流

File由此引申出了 FileInputStream / FileOutputStream / FileReader / FileWriter

Pipe由此引申出了 PipeInputStream / PipeOutputStream / PipeReader / PipeWriter

ByteArray由此引申出了 ByteArrayInputStream / ByteArrayOutputStream / ByteArrayReader / ByteArrayWriter

data是基本数据类型，显然字符没有啥意义（没有编码）由此引申出了 DataInputStream / DataOutputStream

Buffer由此引申出BufferedInputStream / BufferedOutputStream / BufferedReader / BufferedWriter （只有缓冲区达到要求才会和内存或者硬盘交互）

Print由此引申出PrintStream / PrintWriter（打印流，前者只能输出字节，后者可以输出字符）

序列化和反序列化由此引申出ObjectInputStream / ObjectOutputStream 显然按照字符读取对象是没有意义的，所以只有字节流

如何实现对于字符流到字节流的转换呢？
可以使用InputStreamReader和OutputStreamWriter，前者是字节流转字符流，后者是字符流转字节流

CGLIB动态代理。

CGLIB是一个基于ASM的字节码生成库，允许在运行的时候对字节码进行修改和动态生成，CGLIB通过继承方式实现代理

JDK动态代理只能代理实现了接口或者直接代理接口，而CGLIB可以代理任何未实现任何接口的类。由于是子类继承拦截，所以不能代理声明为final类型的类和方法
