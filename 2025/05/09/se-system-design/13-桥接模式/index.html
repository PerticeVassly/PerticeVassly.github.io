桥接模式

桥接模式 （Bridge Pattern）

（Bridge Pattern）将抽象部分与它的实现部分分离，使其都可以独立的变化，是一种对象结构型模式，由称之为柄体（Handle and Body）模式或者接口（Interface）模式

Bridge Pattern: Decouple an abstraction from its implementation so that the two can vary independently.

模式结构

classDiagram
    class Client {
    }

    class Abstraction {
      + operation()
    }

    class Implementor {
      + operationImpl()
    }

    class RefinedAbstraction {
      + operation()
    }

    class ConcreteImplementorA {
      + operationImpl()
    }

    class ConcreteImplementorB {
      + operationImpl()
    }

    Client --> Abstraction
    Abstraction <|-- RefinedAbstraction
    Implementor <|-- ConcreteImplementorA
    Implementor <|-- ConcreteImplementorB
    Abstraction o--> Implementor : implementor

包括一个抽象类 Abstraction

一个扩充抽象类 RefinedAbstraction

实现类接口 Implementor

具体实现类 ConcreteImplementor

可以看到桥接模式的上半部分其实和一个策略模式非常相似，也是一个组合的关系，但是相比之下，桥接模式的context也是一个抽象类，而不是一个具体的类，这个抽象使得可以进一步扩展出一个维度

优缺点

优点：

分离抽象接口及其实现部分

当然上述方式也可以用多继承的形式来实现，但是除了java这样的语言不支持多继承以外，排列组合产生的类依然非常多，组合的方式更加灵活

提高了系统的可扩展性

实现细节对于客户透明，可以对用户隐藏实现细节

缺点：

增加了系统和理解和设计难度

而且只有两个独立变化的维度的时候可以使用，有些许局限？当然可以使用多个组合的形式

适用环境

不希望再两个变化的维度上产生继承联系

希望抽象化角色和实现化角色可以以继承的方式独立扩展而互不影响

一个类存在两个独立变化的维度

设计需要独立管理这两个变化的维度

不希望引入大量的子类

模式应用

Java语言就是一个很好的例子。

通过虚拟机的抽象实现了不同平台的桥接

GUI的桥接，具体渲染使用不同平台，但是抽象图片逻辑是相同的

JDBC桥接模式，实际上一个程序和运行数据库之间的桥接

模式扩展

适配器与桥接模式

系统初步设计的时候，当发现系统与已有类无法协同工作的时候，可以采用适配器模式
