TDD

通过先写测试再写实现的逆向流程驱动代码设计

三阶段循环：红 -> 绿 -> 重构

终极目标：创建可测试可维护最小化的代码

标准流程（红绿重构循环）

  1. 编写失败测试

  2. 快速实现通过

  3. 逐步泛化

  4. 持续重构

设计

xp倡导演进式设计而非计划设计

xp通过实践“压平”变更曲线

核心使能实践的三位一体：Testing -> Continuous Integration -> Refactoring

赋能实践的核心是测试和持续集成

简单设计的四大准则

通过所有测试

消除重复

清晰表达意图

最小化元素数量

YAGNI (You Aren’t Gonna Need It)

包含未经验证的灵活性

零成本的超前开发

边际成本陷阱

提前N个周期开发的成本往往会比实际需求到来时的成本高
YAGNI的核心是：不是反对设计，而是要用经济学证据驱动每行代码的投资决策

可逆性

通过降低流程中的不可逆性来应对复杂性

xp仍然需要以下设计

持续保持对于代码简洁清晰的渴望

通过重构自信改变代码能力

精通模式

遇见变化的眼光

运用代码，图标及最重要的–对话，向相关人员传达设计

持续集成CI

团队成员频繁的（通常每天至少一次）将其代码更改集成到共享的代码库主线中

任何个体开发者偏离主线的实践都不会超过几小时，意味着每次集成涉及的代码量都相对较小，从而显著降低了合并冲突的可能性和复杂性
