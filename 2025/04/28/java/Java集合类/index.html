Java集合层次结构

Collection 是最顶层接口

旗下有三个主要子接口，分别是

Set，List，Queue

Set：不允许重复元素，旗下有一个子接口SortedSet，表示有序的集合，SortedSet有一个实现TreeSet，HashSet直接实现Set接口，LinkedHashSet继承HashSet（为什么不搞一个UnsortedSet，因为不提供排序获取的方法默认就是一个unsortedSet，要遵循最小接口原则）

List：允许重复元素。旗下包含ArrayList，LinkedList，Vector。只有vector是线程安全的，ArrayList和LinkedList都不是线程安全的。同时Stack类继承自vector。

Qeque：一个队列，按照一定的优先级获取元素，Deque是一个双端队列，LinkedList也实现了Deque接口。Deque相比Qeque变成了一个双端队列，ArrayDeque和LinkedBlockingDeque实现了Deque接口。可以看出LinkedList除了有List按照index获取元素的方式外，还可以使用Deque的方式获取元素。

Map：一个键值对的数据（和集合是并列的），显然也有SortedMap接口，TreeMap就是针对这个接口的实现。HashTable是直接实现的Map接口，HashMap也是实现的Map接口，旗下还有一个LinkedHashMap

一些简单的底层数据架构汇总

linkedlist：1.6之前是循环链表，1.7取消了循环

HashSet的底层采用了一个HashMap来保存元素

LinkedHashSet的底层采用了一个LinkedHashMap来保存元素

TreeeSet使用的是红黑树（效率比问题，AVL旋转消耗太大了）

ArrayDeque，可扩容动态双向数组

Map

HashMap，一个数组 + 链表，首先优先进行数组扩容，如果扩容后还是链表长度大于8，则会转为红黑树。

LinkedHashMap，底层使用了一个双向链表 + HashMap。维护了键值对的插入顺序

但是TreeMap在排序等等方面还是回比LinkedHashMap要快一点。

HashTable是数组加链表组成的

保证线程安全要用ConcurrentHashMap

随着Java并发编程的发展，vector和stack已经被淘汰，推荐使用并发集合类ConcurrentHashMap和CopyOnWriteArrayList来替代。

ArrayList中可以为null，但是这样代码那一维护

ArrayList是对的

应为Linkedlist实际上只有头尾操作，其他操作都需要遍历，而且实际过程中还有大量的读取。LinkedList不支持读取操作。

RandomAccess接口

实际上这个接口啥都没有，只是一个标识

fail-fast，fail-safe

fail-fast：在多线程环境下，使用迭代器遍历集合时，如果在遍历过程中对集合进行了修改，则会抛出ConcurrentModificationException异常。实际上不是严格通过并发实现的

fail-safe：例如copyOnwrite可以保证在修改操作的时候复制出一份快照，快照完成以后，再把这个copyonwrite底层数组引用指向这个新的数组空间。安全失败的意思？？？

Comparable 和 Comparator的区别

一个是单排序，一个是多排序。

后者的实现希望可以将数据逻辑和排序逻辑分开

BlockingQueue

继承自Queue。当队列没有元素的时候一致阻塞，直到有元素；还支持如果队列已满，一致等待队列可以放入新元素的时候再放入。

java中有这样几种实现

  1. ArrayBlockingQueue。使用数组实现有界阻塞队列

  2. LinkedBlokcingQueue。单个链表实现

  3. PriorityBlockingQueue，单链表

4， SynchronousQueue同步队列，不存储元素的阻塞队列，每次插入都必须等待删除，常用于线程之间的直接传递数据

  1. DelayQueue：延迟队列，其中元素只有到了其指定的延迟时间，才能从队列中出队

上述ArrayBlokcingQueue和LinkedBlockingQueue是线程安全的。

HashMap和HashTable

前者是非线程安全的，后者是线程安全的，因为里面的方法都用synchronized上了锁。

补充一点HasHMap的key可以储存null的

HashMap是直接按照2的幂次来做的，因为hash一般使用int来标识，大概有40亿个，很难碰撞。当然这个40亿不可能是一个数组

HashTable是乘二

而且HashMap 1.8之后得到了加强，链表可以转红黑树

HashMap和TreeMap的区别

实现NavigableMap接口让TreeMap有了对于集合内元素的搜索能力。包括定向搜索，子集操作，逆序试图，边界操作等等

注意hashmap会存在一个扩容的问题，为了保持hash重新分布，hash函数实际上是hashcode 加 （n - 1） & hash

阈值8和64

64显然是因为小数组中扩容成本低，优先扩容，

正常情况下链表长度大于8的概率很低，所以当链表长度大于8，说明已经有大量数据了，此时红黑树优势显现

HashMap多线程操作导致死循环问题

原因主要在于多个线程同时操作链表，容易有bug

1.7以后决定使用尾查法，但是显然，虽然不会有头插法这样大量丢失情况，尾插法还是会有数据覆盖问题

ConcurrentHashMap和HashTable的区别

底层采用分段数组 + 链表。

所谓分段，就是指对于整个桶数组进行了分割分段，每一把锁只锁容器其中的一部分数据，多线程访问容器里不同的数据段的数据，不存在锁竞争，可以稍微提高一点数据

1.8以后直接使用Node数据 + sychronize和乐观锁CAS来进行操作，锁实际上加到每个node上去了

HashTable线程安全的，使用sychronized来访问效率比较低

ConcurrentHashMap和HashMap一样不允许key为null，但是相比较HashMap，ConcurrentHashMap不允许value为null（无法判断是key不存在还是因为值本来就是null）其实hashmap允许null的，但是hashmap的compute又通过lambda实现了自动删除值为null的key的方法，所以看起来好像不允许一样

如果你非要NULL ，只能使用public static final Object NULL = new Object();来代替null

ConcurrentHashMap不一定保证所有复合操作的原子性，这里的复合操作指的是（put，get，remove，containskey等）操作，例如可以先判断某个键是否存在，然后根据结果进行查询和更新put（key， value），这样的操作是调用端应该保证并发安全的。

但是其也提供了一些封装好的复合多操作，putIfAbsent，compute，computeIfPresent，merge等方法来保证原子性操作。
