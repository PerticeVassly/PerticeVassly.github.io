观察者模式

观察者模式 (Observer Pattern)

定义对象之间的一种一对多的依赖关系，使得当一个对象改变状态时，其相关的依赖对象都能得到通知并被自动更新。观察者模式又叫做发布-订阅模式（Publish-Subscribe Pattern），模型-视图（Model/View）模式，源-监听器（Source-Listener）模式。观察者模式是一种对象行为模式

Observer Pattern: Define a one-to-many dependency between objects so that when one object changes state, all its dependents are notified and updated automatically.

模式结构

    class Subject{
        +attach(Observer)
        +detach(Observer)

        +update()

    Subject <|-- ConcreteSubject
    Observer <|-- ConcreteObserver
    Subject --> Observer : observers
    ConcreteObserver --> ConcreteSubject : subject

这里实际上是双向持有引用，subject通知，observer来取

当然上述实现也有一定问题。比如说每次被观察对象都传输了所有数据，而不考虑观察者是不是需要。而且每次被观察者数据变动都会调用notify(), 被观察者无法自主决定速率。应该改成上述模式结构中互相持有引用的形式

优缺点

优点

实现类表现层和数据逻辑层的分离

在观察目标和数据逻辑层之间建立了一个抽象的耦合

支持广播通信

符合 “开闭原则”

缺点

如果一个观察目标有很多直接和间接的观察者，通知所有的观察者比较耗时

如果观察目标和观察者之间有循环依赖关系，可能会导致系统崩溃

观察者模式没有相应的机制让观察者知道所观察的目标对象是怎么发生变化的，仅仅知道观察目标发生了变化

Java中的观察者模式

classDiagram
    class Observable {
        - changed : boolean
        - obs : Vector
        + Observable()
        + addObserver(Observer o) : void
        + deleteObserver(Observer o) : void
        + notifyObservers() : void
        + notifyObservers(Object arg) : void
        + deleteObservers() : void
        + setChanged() : void
        + clearChanged() : void
        + hasChanged() : boolean
        + countObservers() : int
    }

    class Observer {
        + update(Observable o, Object arg) : void
    }

    ConcreteObservable --|> Observable
    ConcreteObserver --|> Observer
    Observable --> Observer : obs

为什么会有setChanged()? 被观察者是否发送notify还取取决于setChanged()得状态

void update(Observable o, Object arg); 前者o可以提供get方法来让observer自取需要得数据而非全部发送。arg用于传输一些额外定制的而数据, 这里和上述模式结构有略微的不同，模式结构中是直接将subject作为一个成员变量在其中的，这里是作为参数，效果是一样的。

谁触发更新

目标和它的观察者依赖于通知机制来保持一致。

  • 由目标对象的状态设定操作在改变目标对象的状态之后自动调用Notify, 缺点是多个连续的操作会产生多次连续的更新

  • 让客户负责在适当的时候调用Notify。这样做的有点事客户可以在一系列的状态改变完成后再一次性地触发更新，避免了不必要的中间更新。缺点是给客户增加了触发跟新的责任，客户端可能会忘记调用Notify

封装复杂的更新语义

当目标和观察者之间的依赖关系特别复杂的时候，可能需要一个维护这些关系的对象。将这样的对象称之为更改管理器 （Change Manager）

尽量减少观察者反映其目标的状态变化所需的工作量。例如，如果一个操作设计对于几个相互依赖的目标进行改动，就必须保证仅在所有目录都更改完毕以后，才一次性通知他们的观察者，而不是每个目标都通知

classDiagram
    class ChangeManager {
        - mapping : Subject-Observer
        Register(Subject, Observer)
        Unregister(Subject, Observer)
        Notify()
    }

    class SimpleChangeManager {
        + Register(Subject, Observer)
        + Unregister(Subject, Observer)
        + Notify()     
    }

    class DAGChangeManager {
        + Register(Observer)
        + Unregister(Observer)
        + Notify()
    }

    class Observer {
        + Update(Subject)
    }

    class Subject {
        + Attach(Observer)
        + Detach(Observer)
        + Notify()
    }

    Subject --|> ChangeManager : chman
    ChangeManager --> Subject : subjects
    ChangeManager --> Observer : observers

    SimpleChangeManager --|> ChangeManager

    DAGChangeManager --|> ChangeManager

其中观察目标就是Model，观察者就是视图，控制器就是观察者和模型之间的中介
