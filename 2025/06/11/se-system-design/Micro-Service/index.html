Micro Service

一些常见的架构

单体架构，分层架构，SOA架构，微服务架构

单体架构

全部功能被集成在一起作为一个单一的单元；

主要依靠进程间通信和方法间调用；一般是单一数据库；易于开发，部署，测试，伸缩，但是过于复杂，容易出错，可靠性差，部署周期长

分层架构

关注分离原则。表现层，业务层，持久层，数据层。每一层都有特定职能。

面向服务架构（SOA架构）

一个分布式组件的集合，这些组件为其他组件提供服务（Provider），或者消费其他组件所提供的服务（Consumer）；

各个组件由一个企业服务总线（ESB）提供支持环境。

服务编排引擎（Orchestration Engine）， 可以根据预先定义的脚本对服务消费者与服务提供者之间的交互进行指挥；

SOA的特点

高内聚/松耦合

良好的互操作性

模块化，高重用性

服务动态识别，注册，调用

系统复杂性提高

难以测试验证

各独立服务的演化不可控

中间件易成为性能瓶颈

SOA实现原则

服务解耦：关系最小化，只知道接口

服务契约：各个服务按照文档契约行事

服务封装：除了服务契约所表述的内容，服务将对外部隐藏实现逻辑

服务重用：逻辑分布

服务组合：多个服务组合完成任务

服务自治：服务自行管理封装的逻辑

服务无状态：服务将一个活动所需的保存的咨询最小化

微服务架构

把应用程序功能性分解为一组服务的架构风格，每一个服务都是由一组专注，内聚的功能职责组成

微服务架构六大的特征

服务组件化

  • 服务可以独立替换或者升级的软件单元

  • 服务是进程外的组件，通过Web或者RPC通信

  • 微服务实现组件化，分解成服务

  • 服务可以独立部署，产生更明确的发布接口

围绕业务能力组织

  • 传统软件开发存在跨团队成本

  • 围绕业务能力组织，其团队都是跨职能的

  • 采用产品开发模式，而非项目模式，开发团队负责软件的整个产品流程

内聚和解耦

  • 内聚指的是单一职责，有各自的领域逻辑

  • 解耦指的是为服务间尽量减少直接依赖，独立自治

去中心化

  • 去中心化治理，只需要确定服务之间的接口和运维规范，其余技术栈可以自行决定

  • 每个微服务选择和管理自己的数据库，自己存储

  • 强调服务间无事务合作，相应的修复需要更强大的代价

基础设施自动化

  • CICD，自动化测试，自动化部署

服务设计与演进

  • 高可用

  • 演进式设计

微服务的模式和模式语言

模式：针对特定上下文中发生的问题的可重用解决方案

Context：问题特定的场景

Problems：需要解决的问题

Forces：需求，优先级和其他约束

Solution：可重用的解决方案

Consequences：采用模式后的结果

Related patterns： 与其他模式之间的关系？Predecessor，Successor…

微服务架构模式语言：一组模式，帮助解决微服务架构中的问题（拆分问题，数据问题，通信问题，部署问题等）

微服务架构拆分模式

关键步骤

  1. 定义系统操作

    1. 分析用户故事中频繁出现的名词，创建领域模型
    2. 分析用户故事和场景中的动词，确定系统操作

  2. 定义微服务（围绕业务概念而非技术）、

    • 根据业务能力进行服务拆分: 架构稳定，且开发团队围绕交付业务价值而非技术特性进行组织，高内聚松耦合；但是分析业务能力比较困难
    • 根据子域进行服务拆分：同上，消除上帝类和优化拆分；如何识别子域比较困难
    • 根据动静态调用关系来进行服务拆分：比较适用于遗留系统，保持调用关系不受大的影响

  3. 定义服务API和协作方式

    1. 将系统操作分配给服务
      • 确定请求的初始入口服务
      • 映射不清晰的服务
      • 其他情况将分配给具有处理它所需信息的服务
    2. 确定支持协作的API
      • 某些操作可以由单个服务处理
      • 某些操作跨过整个服务
      • 进程间通信技术

微服务架构部署模式

单主机部署多个服务实例

  • 资源利用率高

  • 版本冲突，隔离性差，版本依赖冲突

单主机部署单个服务实例

  • 服务彼此隔离，无版本冲突

  • 资源利用率低级

服务部署到虚拟机

  • 扩展方便，服务间隔离，VM能够对服务资源消耗更加精确控制

  • 资源利用率低，部署慢，额外开销

服务部署到容器

  • 同上，且启动速度很快

  • 增加容器管理工作，Docker基础设施不如虚拟机基础设施

服务部署到平台（k8s集群）

无服务器部署

  • 集成简单，且不需要自己管理任务，弹性申诉

  • 长尾延迟，基于有限事件与请求的编程模型

微服务架构可观测性模式（运维）

日志聚合

使用集中式日志记录服务聚合来自每个服务实例的日志，用户可搜索和分析日志

  • 处理大量的日志需要大量的基础设施

审计日志

业务逻辑中添加审计日志代码

  • 提供了用户操作记录

  • 审计代码和业务逻辑交织

应用程序指标模式

检测服务以收集有关各个操作的统计信息，在集中式指标服务中聚合指标，提供报告和警报。

  • 提供对应用程序行为的深入洞察

  • 指标代码与业务逻辑交织在一起，使其更加复杂，聚合指标可能需要大量的基础设施

分布式追踪模式

为每个外部请求分配一个唯一的外部请求 ID；并在提供可视化和分析的集中式服务器中记录请求如何从一个服务流向下一个服务；记录在集中服务中处理外部请求时执行的请求和操作的信息（例如开始时间、结束时间）

  • 使开发人员能够通过在聚合日志中搜索其外部请求 ID来查看单个请求是如何处理的

  • 聚合和存储追踪数据可能需要大量的基础设施

异常跟踪模式

向集中式异常跟踪服务报告所有异常，该服务聚合和跟踪异常并通知开发人员。

  • 更容易查看异常并跟踪其解决方案

  • 异常跟踪服务是额外的基础设施

健康检查API模式

服务具有/health返回服务健康状况的健康检查 API 端点（实例Spring Boot Actuator），执行检查；监控服务、服务注册表或负载均衡可以定期“ping”调用端点来检查服务实例的健康状况

  • 定期测试服务实例的健康状况

  • 服务实例可能在健康检查之间失败
