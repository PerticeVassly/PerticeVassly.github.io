DevOps

背景

开发和运维各自为政，缺乏协作；开发专注新功能，运维负责部署维护

DevOps

特点：

DevOps 以敏捷项目管理技术和微服务支持为中心，通过基于版本控制标准的自动化来处理整个软件开发生命周期。

Git 是DevOps 中最流行的版本控制解决方案，DevOps 还包括管理软件生命周期、自动化代码测试、容器编排、云托管和数据分析的CI/CD 要求

好处：

敏捷团队项目管理：增强网站和移动应用软件开发的管理。

软件开发过程的优化：通过持续集成和持续交付(CI/CD) 功能实现。借助CI/CD，公司可以通过代码更改快速推出新的软件功能，从而将新的创新推向市场。使用自动版本控制系统和容器简化了Web服务器代码或应用程序脚本的升级。

促进协作：Git 允许开发人员在具有订单项回滚能力的团队中进行协作。

通过自动化提高效率：CI/CD 通过企业编程工具、IDE 和第三方实用程序支持自动化代码测试。DevOps

使采用它来管理软件开发生命周期的公司能够更好地自动化数据中心流程、Web 服务器配置、数据库管理、知识共享、部署调度和商业智能

DevOps核心实践与技术体系

不仅仅是理念：更是一系列核心实践和技术工具和体系化应用，贯穿软件生命周期（规划，编码，构建，测试，发布，部署，运行，监控，反馈）

持续集成（CI-Continuous Integration）

持续集成（Constinuous Integration，CI）：要求开发者频繁将代码变更合并到主干分支，触发自动化构建和测试。

持续交付（CD-Continuous Delivery）

持续交付（CD-Continuous Delivery）：将CI产出的构建自动部署到测试/预生产环境

持续部署（CD-Continuous Deployment）

持续部署（Continuous Deployment）: 持续交付的极致，代码通过自动化流程验证后直接部署到生产环境

自动化测试

自动化测试：“测试门禁” 只有通过所有自动测试的构建才可以部署，前移质量控制

基础设施即代码（IaC-Infrastructure as Code）：

  • 手动将运维操作转化为可编程脚本/配置

  • 实践：以声明式配置文件定义所需要的基础架构环境，自动部署和变更管理

  • 效果：确保环境配置一致性和可重复性，避免环境漂移

  • 跨云标准化：如Terraform可用同一套脚本在不同云厂商创建相同架构

  • 版本控制与审核：基础设施配置脚本可存入版本库，变更可代码评审，CI/CD测试部署

自动化配置

容器化技术

推动一次构建，到处运行

容器编排

容器编排：自动部署，管理容器集群，提供服务发现，弹性伸缩等等

监控反馈

持续监控和反馈

实时监控：Devops生命周期后半段强调对应用运行状态的实时监控，例如Prometheus，Grafana等

实践：收集系统性能指标，日志，用户行为数据等

持续反馈：不仅是故障响应，更将正常运行数据用于改进

总结

Devops的核心实践是形成CI-CD-CF的闭环。高度自动化使其特征，协作和透明是关键。依赖工具链支撑，更重要是将工具和时间融合合成文化和流程，实现更快交付可靠的软件

Devops与敏捷

敏捷 （Agile）

核心：项目和产品开发方法学，提高开发过程灵活性和响应速度

关注点：开发团队内部效率

目标：“如何更有效地开发软件”

Devops

核心：扩展敏捷理念，返回从开发延申到软件交付和运维全生命收起

关注点：开发与运维协作，打破壁垒，实现从开发到部署的持续流动

目标：“如何更有效地交付软件”

敏捷提供了快速交付增量功能的框架，Devops确保功能顺畅，安全部署上线

敏捷遵循Scrum/Kanban，短迭代交付增量；Devops在敏捷产出基础上引入CICD，自动化部署，监控等等，快速推向生产

Devops与CICD

CICD是技术实践，Devops是文化和理念。CICD是Devops的核心实践之一

没有CICD很难实现DevOps。CICD仅仅是关注代码构建，测试，部署自动化。Devops涵盖人，流程工具范围更广，包括组织和文化

Devops与云原生

云原生：强调利用云计算优势设计部署应用的架构理念。强调应用如何构建运行以适应云环境弹性和分布式特性

云原生关注架构，Devops关注流程

DevSecOps

DevSecOps：将安全融入DevOps流程中，强调安全是每个环节的责任，而不是单独的一个环节 DevSecOps = Dev + Sec + Ops

常用DevOps术语表

蓝绿部署（Blue-Green Deployment）

定义：零停机部署。维护两套环境：蓝（当前生产），绿（部署新版）

流程：验证绿环境正常后，流量从蓝切换到绿

优点在于切换快，易回退。但是一次性切换有风险

使用：要求短暂无缝切换场景

金丝雀发布（Canary Release）

流程：仅仅将一小部分用户/流量切换到最新版本，其余仍然用旧版。观察新版运行情况，

优缺点：更加安全可控，常常与A/B测试结合使用

适用：风险最低的发布策略

MTTR（Mean Time to Recovery/Restore）

定义：平均恢复时间

意义：Devops度量中反映发布稳定性和运维效率的关键指标。

DORA(DevOps Research and Assessment) 指标体系

  • 部署频率

  • 前置变更时间（代码提交到代码成果部署到生产环境所需的时间，也即开发人员开始写代码到客户真正获得此变更的间隔时长）

  • 变更失败率（部署到生产环境的代码变更引发故障或问题，导致需要紧急修复，回滚或者出现服务降级的情况）

  • 平均恢复时间（MTTR）

前两者体现了交付的速度和效率

后两者体现了交付的质量和稳定性
