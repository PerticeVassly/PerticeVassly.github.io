装饰模式

装饰模式 （Decorator Pattern）

Decorator Pattern，动态地给一个对象增加一些额外地职责，就增加对象功能来说，装饰模式比生成子类实现更为灵活。其别名也可以称之为包装器模式（Wrapper），这里和适配器地名字相同。但是适用于不同地场合。根据翻译地不同，也有人称之为油漆工模式，是一种对象结构型模式。

Decorator Pattern: Attach additional responsibilities to an object dynamically. Decorators provide a flexible alternative to subclassing for extending functionality.

模式结构

classDiagram
  class Component {
    + operation()
  }
  class ConcreteComponent {
    + operation()
  }
  class Decorator {
    + operation()
    
  }
  class ConcreteDecoratorA {
    - addedState
    + operation()
  }
  class ConcreteDecoratorB {
    + operation() :
    + addedBehavior()
  }

  Component <|-- ConcreteComponent
  Component <|-- Decorator
  Decorator <|-- ConcreteDecoratorA
  Decorator <|-- ConcreteDecoratorB
  Decorator o--> Component

Decorater 中的 operation 实际上就是调用component的 operation 方法

ConcreteDecoratorB 的 operation可以 super.operation addedBehavior

包含以下角色

Component：抽象构件

ConcreteComponent：具体构件

Decorator：抽象装饰类

ConcreteDecoratorA：具体装饰类A

模式分析

复用 + 增加职责的方式

与继承关系相比，关联关系（组合）的主要优势在于不会破坏类的封装性，而且继承是一种耦合度比较大的静态关系，无法在程序运行时动态扩展。关联关系具有较好的松耦合性，使得系统更加容易维护，缺点是比继承关系要创造更多的对象

以对客户透明的方式给一个对象附加更多的责任，装饰模式可以在不需要创造更多子类的情况下，将对象的功能扩展。

意思是将一个继承复用转化为组合复用，二者实现相同的接口实现统一依赖。具体的装饰类决定要加强的功能是什么。这其实有点像一种代理模式？虽然代理模式还包含了调用拦截这样的功能，但是实际上可以看成是对方法的一个增强？

优缺点

优点

装饰模式（组合）和继承关系的目的都是复用+扩展对象的功能，但是装饰模式由于使用的是组合，提供了更多的灵活性

可以通过一个动态的方式来扩展一个对象的功能，可以在要运行时选择不同的装饰器从而实现不同的行为

通过使用不同的具体装饰类和这些装饰类的排列组合，可以创造出狠多不同行为的组合

具体构建类和具体装饰类可以独立变化。这里有点像桥接模式，但是桥接模式的两个接口时分离的，而这里是统一的以提供复用关系

缺点

将产生很多小对象，这些小对象的区别静静在于其相互连接的方式不同，还有很多具体的装饰类，增大学习和理解的难度

更加容易出错，排错更加苦难，尤其是装饰模式可以多级别装饰

适用场景

动态透明的方式给单个对象添加职责

动态给一个对象增加功能，这些功能也可以动态撤销

当不能采用继承的方式进行扩展的时候或者继承不太合理的时候

模式应用

java io

InputStream下辖FileInputStream ByteArrayInputStream 等等具体实现类

同时还有一个FilterInputStream类同样继承了InputStream类，但是其内部构建的是一个组合形形式复用的InputStream，（其实到这里就可以直接添加新功能了，毕竟只是继承接口，但是为了更好的封装），再创建新子类Buffered, Data等进一步添加功能

模式扩展

装饰器模式的简化需要注意的问题

装饰类的接口必须和被装饰类的接口保持相同，这样如论是装饰前还是装时候的对象都可以保持一致，虽然由于是继承，所以其实可以不一致，增加新功能

具体构建类应该是一个轻类

如果只有一个具体构建类但是没有抽象构建类（即不需要针对不同的具体构建类进行装饰），那么抽象装饰类可以作为具体构建类的直接子类（然后重写其全部方法？）但是这样还是触及到了继承复用的缺点了

如果只有一个具体装饰类，可以直接和抽象装饰类合并，毕竟本身也没有任何办法限制抽象装饰类只实现抽象构件类的方法，所以你可以很暴力直接将抽象装饰类变得不抽象，在其上直接增强。也可以稍微优雅一点，在划分子类的时候增强。但是实际上约束取决于客户端，客户端采用透明还是非透明的方式决定了，你新增的这些方法到底能不能被使用

透明装饰模式：客户端完全基于抽象编程，客户端完全不知道具体构件类和装饰类的存在，所有的操作都是基于抽象构件类进行的，此时装饰仅仅限于功能增强

半透明装饰模式：客户端知道具体构建类和装饰类的存在，所有的操作都是基于抽象构建类进行的，但是客户端可以通过具体构建类来获取具体构建类的功能，即获取到增加的方法。

补充一下，如果抽象装饰类没有继承抽象构建类呢？那其实相当于客户端想如何用构建类就如何使用了，退化成了一个策略模式了，甚至不能算是适配器模式，因为根本没有任何规定好的接口，直接退化成了一个策略模式了（组合模式！=组合）
